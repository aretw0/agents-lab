# Agent Quality Review Rubric — May 2026

> Scope: patches produced by simple worker agents in agents-lab.  
> Status: research draft (v0.8).

---

## 1. Scope

This rubric applies to **canary and routine patches** generated by simple worker agents (non-orchestrator subagents) operating within agents-lab. It does not cover:

- Orchestrator logic or scheduler changes
- Provider / credential configuration
- CI/GitHub Actions pipelines
- Publish or remote/offload flows
- Protected parked tasks

Each patch is reviewed against the sections below before merge. A patch that touches out-of-scope files must be escalated; it cannot be self-reviewed with this rubric.

---

## 2. Validation

A patch passes validation when **all** of the following are true:

| # | Check | How to verify |
|---|-------|---------------|
| V1 | Only allowed files are modified | `git diff --name-only` matches the task's declared file scope |
| V2 | No runtime code altered | Diff contains no changes outside `docs/`, `scripts/` (read-only helpers), or explicitly listed source files |
| V3 | File is well-formed Markdown / valid syntax | `markdownlint` or equivalent linter passes; no broken front-matter |
| V4 | Content satisfies stated goal | Reviewer confirms the artifact addresses the task description |
| V5 | No secrets or credentials introduced | `git diff` search for common secret patterns returns nothing |
| V6 | Agent stopped at the declared slice boundary | Task log shows a single commit-free work unit with a summary + evidence |

---

## 3. Rollback

Because agents operate on **git worktrees without committing**, rollback is low-cost:

1. If the patch is rejected pre-commit, discard the worktree: `git checkout -- <file>` or delete the worktree branch.
2. If the patch was committed by mistake, revert: `git revert <sha>` — do not force-push.
3. Preserve the agent's summary log in the task ticket before discarding, for post-mortem reference.

Rollback is considered complete when `git status` returns clean and the main branch is unaffected.

---

## 4. Reject Conditions

**Reject immediately (no discussion needed) when:**

- The patch touches any out-of-scope file (see §1).
- The agent committed without explicit human approval.
- A secret, credential, or environment variable is introduced or exposed.
- The patch modifies CI, GitHub Actions, scheduler, or provider configuration.
- The content is empty, placeholder-only, or copy-pasted boilerplate with no task-specific substance.
- The agent looped or produced duplicate artifacts beyond the declared slice.

**Reject and request revision when:**

- Validation checks V1–V6 pass but content quality is below the acceptance threshold (see §5).
- The rubric section is present but hollow (e.g., a table with no meaningful rows).
- Formatting renders broken in standard Markdown viewers.

---

## 5. Acceptance Threshold

A patch is accepted when it clears **all hard gates** (§4 reject conditions absent) and meets **≥ 4 of 5** quality signals:

| Signal | Criterion |
|--------|-----------|
| Q1 — Completeness | All required sections/markers present and non-trivial |
| Q2 — Precision | Claims are specific; no vague language where specifics are available |
| Q3 — Conciseness | No padding; each paragraph earns its place |
| Q4 — Correctness | No factual errors detectable by a domain-aware reviewer |
| Q5 — Scope discipline | Agent demonstrably stopped at the slice boundary |

A patch that passes all 5 signals but fails a hard gate is still rejected.

---

## 6. Review Burden

The goal is to keep review lightweight for local-safe canary patches.

**Target effort per patch:** ≤ 10 minutes for a reviewer familiar with the project.

Practices that keep burden low:

- **Small slices**: one file, one goal per agent run. Multi-file patches automatically increase review time and risk.
- **Clear evidence**: agent returns a diff summary + the file path changed. Reviewer need not reconstruct what happened.
- **Pre-validated scope**: the task ticket declares the allowed file before the agent runs. Scope check (V1) becomes a 30-second read.
- **No commit**: agent never commits. Reviewer controls when work enters history.

If a patch requires > 10 minutes to review, that is a signal the slice was too large or the task description was underspecified — adjust the task, not the rubric.
